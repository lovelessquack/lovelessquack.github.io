# 项目现状梳理

## 文档目的

这份文档用于快速恢复对当前仓库的理解，帮助后续继续开发或整理时先看清现状、边界和风险。

这不是重构方案，也不是任务拆解文档。重点是回答三个问题：

- 这个仓库现在是什么
- 它已经能做什么
- 下一步最值得先处理什么

## 理解摘要

- 这是一个纯静态站点仓库，没有构建流程，也没有自动化测试。
- 根入口是 `index.html`，两个主要功能页位于 `html/`。
- `html/card.html` 是文字卡片生成器，`html/chat.html` 是聊天截图生成器。
- 共享样式和逻辑集中在 `static/`，其中 `index.css` 承担了大部分页面样式。
- 页面依赖第三方字体、头像服务和 `html2canvas`，所以运行和导出都受网络与浏览器安全上下文影响。
- 当前最明显的问题不是功能缺失，而是编码异常和敏感配置硬编码。

## 页面职责

### `index.html`

当前职责很简单：

- 提供顶部导航
- 作为工具落地页
- 引导进入卡片页和聊天页

从实际代码看，这一页更像入口壳，而不是独立功能页。

### `html/card.html`

这是当前相对复杂的页面，职责包括：

- 输入标题、正文、署名、头像链接、签名
- 切换多种卡片风格
- 实时生成卡片预览
- 用 `html2canvas` 导出 PNG
- 复制图片到剪贴板
- 额外提供“文字入库”动作

它本质上是一个“文字内容 -> 分享图”的前端生成器。

### `html/chat.html`

当前职责包括：

- 输入聊天对象昵称和头像
- 逐条添加左右两侧消息
- 清空聊天记录
- 生成微信风格聊天截图预览
- 用 `html2canvas` 保存或复制图片

它本质上是一个“手动构造聊天记录 -> 导出截图”的生成器。

## 共享资源分工

### `static/index.css`

这个文件承担了明显超出“基础样式”的职责：

- 页面级布局
- 顶部导航
- 表单控件
- 卡片主题样式
- X 风格卡片样式
- 微信聊天截图样式
- toast 和 loading overlay

也就是说，当前样式是“单文件总管式”组织。好处是简单直接，坏处是后续维护时耦合较高。

### `static/index.js`

当前主要负责卡片页的完整交互流程：

- 初始化默认状态和头像
- 绑定表单与按钮事件
- 生成卡片内容
- 根据内容长度调整卡片布局
- 绘制标题高亮和正文虚线装饰
- 调用 `html2canvas` 保存和复制图片
- 将文字 POST 到服务端接口

这是当前仓库里最需要谨慎处理的脚本，因为它同时包含视觉逻辑、导出逻辑和服务端提交逻辑。

### `static/chat.js`

当前主要负责聊天页的交互流程：

- 维护聊天历史状态
- 初始化头像
- 添加左右气泡消息
- 渲染聊天记录
- 更新顶部聊天对象名称
- 保存或复制聊天截图

结构上比 `static/index.js` 简单，功能边界也更清晰。

## 当前运行方式

项目没有构建过程，推荐运行方式仍然是：

```powershell
python -m http.server 8000
```

然后访问：

```text
http://localhost:8000/
```

这样做的原因很明确：

- 剪贴板 API 依赖安全上下文
- `html2canvas` 导出更适合在本地服务下测试
- 第三方资源加载和跨域表现更接近实际部署环境

## 第三方依赖

目前仓库依赖的外部资源主要有：

- Google Fonts
- CDNJS 的 `html2canvas`
- DiceBear 头像服务
- 外部头像图片地址

这意味着项目具备以下特点：

- 离线不可完全运行
- 导出能力会受外链图片和 CORS 影响
- 页面渲染稳定性受外部服务可用性影响

## 当前已确认问题

### 1. 编码异常明显

从 `index.html`、`html/card.html`、`html/chat.html`、`static/index.js`、`static/chat.js`、`static/index.css` 的读取结果看，中文出现大量乱码。

这说明至少存在以下一种或多种情况：

- 文件原始编码与声明不一致
- 历史保存过程中发生过转码损坏
- 局部文件编码统一性已经丢失

这件事的优先级很高，因为在没有确认编码前，继续直接修改中文文案风险很大。

### 2. 敏感配置硬编码在前端

`static/index.js` 中存在：

- 硬编码接口地址 `http://192.168.2.3:9991/api/duanzi`
- 硬编码请求头 `X-API-Key`

这不是普通技术债，而是直接的配置与安全问题：

- 会暴露内部网络信息
- 会扩散密钥使用方式
- 也让前端页面和特定局域网环境强绑定

### 3. 样式集中但耦合偏高

`static/index.css` 同时维护首页、卡片页、聊天页、多种主题和导出辅助样式。

短期没问题，但后续如果继续加页面或新增风格，维护成本会明显上升。

### 4. 页面职责基本清楚，但缺少项目级说明文件

除了 `AGENTS.md` 外，目前仓库没有一个面向开发者自己的项目总览文件。

这也是当前补 `plan.md` 的直接价值所在。

### 5. 手工验收依赖较重

目前没有自动化测试，任何改动都需要手工覆盖：

- 首页导航
- 卡片生成
- 聊天截图生成
- 保存图片
- 复制图片
- 外链头像加载

这会让后续变更成本越来越依赖人工回归。

## 当前状态判断

这个项目目前不属于“架构混乱无法继续”，而是典型的“小型静态工具站已经可用，但需要补基础治理”。

换句话说，现阶段最重要的不是大改功能，而是先把几个会持续放大风险的问题压住：

- 编码
- 敏感配置
- 资源与职责边界

## 建议优先级

### P1：先确认并统一编码

这是所有后续修改的前提。

如果编码问题不先解决，任何中文文案、注释、页面结构调整都有二次损坏风险。

### P2：移除或隔离前端硬编码接口与 API Key

至少要先停止继续扩散这种写法。

即使暂时不重构完整链路，也应该先把这部分从“默认前端公开代码”里剥离出来。

### P3：补最小项目说明和维护约束

当前这份 `plan.md` 已经补上了第一步。

后面可以继续明确：

- 哪些页面还在使用
- `x/` 目录是否保留
- 哪些第三方依赖是必须的

### P4：按页面拆分样式或职责边界

不是立刻重写，而是在后续有改动时逐步把：

- 全站共用样式
- 卡片页专属样式
- 聊天页专属样式

分开管理，避免继续把所有内容塞进一个 CSS 文件。

### P5：整理最小手工验收清单

哪怕不做自动化测试，也建议固定一份最小回归 checklist，至少覆盖：

- 首页导航跳转
- 卡片页生成与导出
- 聊天页生成与导出
- 头像加载
- 剪贴板复制
